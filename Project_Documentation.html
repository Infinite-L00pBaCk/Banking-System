1. Project Overview

This project is a C++-based Banking Management System that simulates real-world banking operations through a console interface. It allows users to create accounts, manage balances, perform deposits/withdrawals/transfers, and view complete transaction history.

All data is persisted to files, meaning accounts and transactions survive between program restarts.

3. OOP Concepts Used

3.1 Encapsulation

All class data members are private. Access is provided through public getter methods and controlled operations.

class BankAccount {
private:
    std::string accountNumber;  // Cannot be accessed directly
    std::string pin;            // Hidden from outside
    double balance;             // Modified only via deposit()/withdraw()
public:
    double getBalance() const;  // Controlled read access
    bool deposit(double amount);  // Controlled write access
};

Why? Prevents external code from setting invalid balances or bypassing PIN checks.

3.2 Abstraction

The Bank class hides the complexity of file I/O, account lookup, and transaction logging. The main.cpp simply calls bank.deposit() without knowing the internal details.

3.3 Modularity (Separation of Concerns)

ClassResponsibility
BankAccountManages one account's data and operations
TransactionRepresents and formats one transaction record
FileManagerHandles all file read/write (persistence layer)
BankOrchestrates everything (business logic layer)
main.cppUser interface layer (menu and input)

3.4 Enum Class

enum class TransactionType {
    DEPOSIT, WITHDRAWAL, TRANSFER_IN, TRANSFER_OUT, ACCOUNT_CREATION
};

Scoped enum (enum class) is used instead of plain enum for type safety — prevents accidental integer comparisons.

4. Core Logic Explained

4.1 Account Creation Flow

  1. User enters full name
  2. User sets a 4-digit PIN (validated: must be exactly 4 digits)
  3. User confirms PIN (must match)
  4. User enters initial deposit (minimum Rs. 500)
  5. System generates unique account number (e.g., PNB-1001)
  6. Account is stored in vector<BankAccount>
  7. Creation transaction is logged to transactions.dat
  8. All accounts are saved to accounts.dat

4.2 Deposit Logic

bool BankAccount::deposit(double amount) {
    if (amount <= 0) → reject (must be positive)
    if (!active)     → reject (account closed)
    balance += amount;
    return true;
}

After a successful deposit, the Bank class creates a Transaction record and appends it to the file.

4.3 Withdrawal Logic

bool BankAccount::withdraw(double amount) {
    if (amount <= 0)      → reject
    if (!active)          → reject
    if (amount > balance) → reject ("Insufficient balance!")
    balance -= amount;
    return true;
}

Key protection: You cannot withdraw more than the available balance.

4.4 Transfer Logic

  1. Verify sender account exists and is active
  2. Verify sender PIN
  3. Verify recipient account exists and is active
  4. Check sender != recipient
  5. Call sender.withdraw(amount)
  6. Call recipient.deposit(amount)
  7. Log TWO transactions: TRANSFER_OUT for sender, TRANSFER_IN for recipient

4.5 Account Number Generation

std::string Bank::generateAccountNumber() {
    return "PNB-" + std::to_string(nextAccountId++);
    // PNB-1001, PNB-1002, PNB-1003, ...
}

On startup, the system scans existing accounts to find the highest ID and continues from there.

5. Data Persistence (File I/O)

5.1 Serialization Format

Data is stored as pipe-delimited (|) text in .dat files.

accounts.dat format:

PNB-1001|Priyam Prakash|1234|5000.00|1|2026-05-07 19:08:00

Fields: AccountNo | Name | PIN | Balance | Active(1/0) | CreationDate

transactions.dat format:

PNB-1001|DEPOSIT|2000.00|7000.00|2026-05-07 19:10:00|Cash deposit

Fields: AccountNo | Type | Amount | BalanceAfter | Timestamp | Description

5.2 Why Pipe-Delimited?

  • Simple to parse with std::getline(stream, field, '|')
  • Human-readable (you can open .dat files in Notepad)
  • No external library needed (no JSON/XML parsers required)
